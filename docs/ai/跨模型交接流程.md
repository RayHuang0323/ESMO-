# 跨模型交接流程

> Claude Code token 不足時，如何安全交接給 Codex（或其他模型）。
> 前置：雙方都先讀 `AGENTS.md`。Codex 專用規則見 `docs/ai/CODEX使用規範.md`。

## 1. Claude token 不足前（交接方）

Claude 應該：

- **停止新增大改**（不要開新的大範圍重構）。
- 跑 `git status --short`、`git diff --stat`。
- 回報：**已完成 / 未完成**清單。
- 標記**未驗證項**（尤其手機 UX / FPS / 視覺 —— Node 證不了的）。
- **不自行亂 commit**；工作區未收尾就照實說明，交給使用者決定。

## 2. 交接給 Codex 前（使用者 / Claude 提供）

- **交接清單指向 `docs/README.md`**（新模型由此決定選讀哪些文件，不重讀完整歷史）。
- 目前 Sprint 目標與範圍
- 最新 commit hash
- `git status` 現況
- 已跑哪些 verifier、結果
- 失敗 / 紅燈項目
- **不可修改區域**（別人正在改的、或高風險契約）
- **只提供任務必要文件**——不要求新模型重新讀完整歷史 Sprint 紀錄。

## 3. Codex 接手流程（接手方）

Codex 應：

- 先讀 `AGENTS.md`、再讀 handoff（00 / 05 / 06 / 08）。
- 先看 `git status` / `git diff`，掌握現況。
- **不重跑全專案 Audit**（浪費 token；handoff 已有現況）。
- **不重寫已完成部分**。
- 先修 **blocking issue**。
- 必要時**先寫 verifier** 再改實作（讓改動有安全網）。

## 4. Codex 完成後（交還）

Codex 回報：

- 修改檔案清單
- 根因（若修 bug）
- 已驗證項（含 exit code / 輸出形狀）
- 未驗證項
- 是否 commit / push
- 下一步 / 交還 Claude 或使用者

## 5. 衝突處理（Codex 發現 Claude 已改一半）

**絕對不要自作主張覆蓋別人的工作。** 若發現工作區已有未完成改動：

- **不 `git reset`**
- **不 `git checkout` 覆蓋**
- **不 `git pull` 覆蓋**
- **不刪未追蹤檔**（`git clean` 禁用）
- 先**回報狀態**（列出未 commit 的檔案與內容摘要）
- **等使用者確認**再動。

## 6. 遇到 verifier 紅燈

- 先判斷：是**這次改動造成**，還是**既有已紅**（見 `CLAUDE.md` 的「已失效 verifier」清單）。
- 若是本次造成 → 修好再繼續，不可刪 verifier 或改斷言硬過。
- 若是斷言本身壞掉（跟不上合法重構）→ 改斷言，**並在 `05_Sprint紀錄.md` 說明原因與證據**。
- 判不出來 → 回報，不硬提交。

## 7. 遇到不確定需求

- **不猜、不擴大範圍**。
- 列出不確定點與可選方案，回報使用者。
- 在得到答覆前，只做範圍明確、可逆的部分。
- **若文件之間有衝突**：依 `docs/README.md` §五 的優先順序初判，並**列出衝突點交由 owner 或使用者裁決**，不自行選較舊文件。
