## 一、架構快照

ESMO 存在兩條血脈。**2026-06-30 起，兩者的定位已正式拍板：**

**B. 主線（Trunk）— 唯一長期維護的架構** — React + Vite + React Three Fiber（three r160）。
- `gameData.js` / `LogicEngine.js` / `useGameStore.js` / `useLocalServer.js` / `MobaView3D.jsx` / `GameView.jsx`
- `data/heroes.js`（2026-06-30 新增）：英雄資料單一真實來源。`ROLE_STATS`/`BASE_HP` 已接入 LogicEngine（取代原本寫死的數值陣列，行為零變更）；`CHAMPIONS_100`（100 位，自 Legacy 逐位元抽出）為參考名單、尚未接入模擬。**註：此為 Hero System 資料化的第一步，非關閉任何編號 TD**（TD-06 base64 與此無關）。
- `data/heroMapping.js`（2026-06-30 新增）：Hero Mapping Layer，將 10 名玩家對應到 `CHAMPIONS_100` 一位英雄（身份映射；純函式、零相依）。`roster` 存於 `useGameStore`、開局由 `useLocalServer` 指派。**戰鬥零影響、LogicEngine 未改動；同樣非關閉任何編號 TD**（Hero System 進度）。
- `package.json` 宣告 three `^0.160.0` + R3F 8 + drei 9 + postprocessing + zustand。
- **這是 ESMO 唯一的長期維護主線。** 所有新開發都在此進行。

**A. Legacy Prototype — 只供參考與功能移植，不再維護** — 為 Claude artifact 沙盒撰寫，使用 three.js r128 舊 API。
- `EsportsGame.jsx`（9,401 行）：功能完整的沙盒成品。含 MOBA／FPS 3D 觀戰、約 22 個經營子系統、`CHAMPIONS_100`/`SKILL_DB`/`PERSONALITY` 等資料與內聯 base64 資產。
- `App.jsx`（803 行）、`EsportsFPS3D.jsx`（1,584 行）：同屬沙盒血脈。
- **定位：功能與資料的「來源庫」。** 後續工作是把這裡的功能逐步**移植**進 B，而非繼續在 A 上開發。A 的 r128 舊 API（`sRGBEncoding` 等）**無需修復**，因為 A 不再被建置為產品。

### 建置進入點
`index.html` → `/src/main.jsx`。**`main.jsx` 不在目前的專案檔內**。依主線決策，它**必須掛載 B 主線（`<GameView/>`）**，不是 legacy 的 `<EsportsGame/>`。見 TD-04 / R0.2-03。

### snapshot 合約（B 血脈的核心，render 只依賴它）
詳見 `INTEGRATION.md`。重點：`LogicEngine.snapshot()` 維持形狀不變時，render 層與 store 一行都不用改。這是這個專案最值得保護的設計。

## 三、已知的反覆性陷阱（接手前必讀）

這些不是待修債務，而是「改某類東西時必須記得做的事」，記錄於此避免重蹈覆轍：

- **Frustum culling**：玩家 mesh 一律 `frustumCulled=false`，否則會閃爍/消失。
- **`MAP_EDGE` 平衡敏感**：任何戰鬥技能模型改動後，需重調 map edge 以回到約 50% T 方勝率。
- **箱子碰撞**：完整箱子碰撞會破壞平衡並卡住角色；只做車輛碰撞才是穩定模式。
- **相機抖動**：`camBlocked` 遮擋迴避會在密集地圖震盪；改用建築淡出透明，不要用相機旋轉迴避。
- **邏輯/渲染純度**：`LogicEngine.js` 不可 import React/Zustand/DOM，否則無法搬 Node.js。
- **內插邊界**：內插是 client 的事；store 存 prev+latest，view 以 60fps lerp，logic 跑 6Hz。
