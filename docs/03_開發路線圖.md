# ESMO 開發路線圖

> 基準日期：2026-06-30｜起始版本：v0.1
> 本檔是滾動式文件，每個版本一節。目前唯一進行中的版本是 **v0.2**。

---

## v0.2 — 地基整理（Foundation）

**版本目標**：不加任何新功能，把基礎架構整理到「適合未來一年以上持續開發」的狀態。
**核心原則**：先讓專案「能被可靠建置、能被 AI 安全接手、單一事實來源明確」，再談擴充。
**驗收門檻**：(1) 能依 `package.json` 一鍵安裝並跑起來；(2) 主線血脈明確，無被覆蓋的重名函式；(3) 文件骨架不再是空殼。

任務 ID 為 `R0.2-xx`，並標註對應的技術債（見 `09_技術債務清單.md`）。
分三階段：**穩定 → 收斂 → 模組化**。標示 `[現在做]` 的請優先；標示 `[延後]` 的附上誠實理由。

### 階段一：穩定（先讓它站穩）

#### R0.2-01　確認 r160 主線、隔離 legacy　`[現在做／已大幅消解]`　對應 TD-01
- **目的**：原任務是「拍板版本並對齊」。**主線已於 2026-06-30 拍板為 B（three r160 + R3F）**，故 `package.json` 的 r160 本來就正確，不需更動。本任務縮為：(1) 確認 B 主線完全不相依任何 A（legacy）檔；(2) 在 README/文件標明 A 為 Legacy Prototype。
- **影響範圍**：僅確認與標記，不改 A 的 r128 程式碼（A 不再建置，舊 API 無需修）。
- **風險**：低（從「中」降下來）。決策已消除原本的版本不確定性。
- **是否值得現在做**：原本是「第一優先」，現在大半已由決策自動完成。剩餘的「確認 B 不相依 A」併入 R0.2-03（進入點落地）一起驗證即可。
- **預估修改檔案**：`README.md`、（驗證）`GameView.jsx` / `main.jsx` 的 import。**A 血脈檔案不動。**

#### R0.2-02　清理被覆蓋的重名函式　`[現在做]`　對應 TD-02、TD-03
- **目的**：移除 `simulateFps`（L351/L3628）與 `HeroAvatar`（L6357/L8086）的重複定義，確保跑的是預期版本。
- **影響範圍**：`EsportsGame.jsx` 兩處頂層宣告。
- **風險**：中。需先確認「現行實際生效的是後定義版」，再決定刪除或合併前者；若前者其實是想用的版本，這正是一個潛藏 bug 的修復點。
- **是否值得現在做**：**是**。重名覆蓋是會造成靜默錯誤的 Critical 債，且修復範圍小、收益明確。
- **預估修改檔案**：`EsportsGame.jsx`。

#### R0.2-03　確認並建立建置進入點　`[現在做]`　對應 TD-04
- **目的**：讓 `index.html → /src/main.jsx` 真的能掛載，確認載入的是哪條血脈。
- **影響範圍**：`src/main.jsx`（可能需新增）、`index.html`。
- **風險**：低。屬於補齊而非改寫。
- **是否值得現在做**：**是**。沒有可靠進入點，前兩項的成果無法驗證。
- **預估修改檔案**：`src/main.jsx`、`index.html`。

#### R0.2-04　Vite 設定檔正名　`[現在做]`　對應 TD-11
- **目的**：把 `vite_config.js` 改為 `vite.config.js`，讓 Vite 能自動載入。
- **影響範圍**：單一檔名。
- **風險**：極低。
- **是否值得現在做**：**是**，順手即可完成；放著會讓正式專案靜默忽略設定。
- **預估修改檔案**：`vite_config.js` → `vite.config.js`。

### 階段二：收斂（讓事實來源唯一）

#### R0.2-05　抽離內聯 base64 資產　`[現在做]`　對應 TD-06
- **目的**：把約 700KB base64 從原始碼移到外部檔（`/public/assets/`），讓單檔回到可被人與 AI 正常閱讀的大小。
- **影響範圍**：`EsportsGame.jsx`（MOBA_MAP/HERO_IMG/CS_MAPS 等）、`App.jsx`、新增資產檔與載入路徑。
- **風險**：中。沙盒不支援外部 import，所以需保留「沙盒用內聯／本地用外部」的雙模式，或確立本地 Vite 為主要開發環境。建議搭配 R0.2-08 一起決定。
- **是否值得現在做**：**是**。這是降低後續所有開發與 AI 協作成本的高槓桿動作。
- **預估修改檔案**：`EsportsGame.jsx`、`App.jsx`、`/public/assets/*`、`index.html`（如需預載）。

#### R0.2-06　填補文件骨架　`[現在做]`　對應 TD-05
- **目的**：讓 `02`～`07` 從 0 bytes 變成最小可用內容，使 README 宣稱的 AI onboarding 路徑成立。
- **影響範圍**：`02_遊戲設計_GDD_.md`、`03`（本檔已建）、`04_更新日誌.md`、`05_英雄資料庫.md`、`06_技能資料庫.md`、`07_待辦事項.md`。
- **風險**：低。寫文件不動程式碼。可先放「結構 + 指向程式碼真實來源（如 `CHAMPIONS_100`、`SKILL_DB`）」的骨架，逐步填實。
- **是否值得現在做**：**是**。這是「給其他 AI / 協作者接手」的前提，與你本次需求直接相關。
- **預估修改檔案**：上述六個 `.md`。

#### R0.2-07　建立 snapshot 合約的單元測試　`[現在做]`　對應 TD-12
- **目的**：為 `LogicEngine.js` 加上基本單元測試（snapshot 形狀、勝率穩定區間、combo 解析、穿牆檢查），把現有的人工驗證 harness 固化成 CI 可跑的測試。
- **影響範圍**：新增 `/tests/`（或 `LogicEngine.test.js`），不改邏輯本身。
- **風險**：低，且能反向保護 R0.2-01 等改動不破壞平衡。
- **是否值得現在做**：**是**。`LogicEngine` 零相依、最好測，投資報酬率最高；有了它，未來一年的數值改動才有安全網。
- **預估修改檔案**：`/tests/LogicEngine.test.js`、`package.json`（test script）。

#### R0.2-08　確立「主線血脈」決策（決策任務，非寫功能）　`✅ 已完成 2026-06-30`　對應 TD-08、TD-07
- **決議**：**GitHub 上的 React + Vite + R3F（three r160）＝ ESMO 唯一主線（Trunk）。** Artifact／沙盒 monolith（`EsportsGame.jsx` 等）定位為 **Legacy Prototype**：只供參考與功能移植，不再作為長期維護架構。
- **目的**：明文寫下 trunk，作為後續所有收斂與版本決策的依據。**這是本次「任務 1」實際完成的項目**——一個零程式碼的決策。
- **影響範圍**：文件層（README、本路線圖、技術債務清單、更新日誌）；不改程式。
- **風險**：低。決策已消除最大的不確定性來源。
- **是否值得現在做**：**是，且已完成**。它一次解決／降級了約五項債（TD-08 解決；TD-01/02/03/07/09 降為 legacy 關注），印證「先決策再動程式」是對的。
- **已修改檔案**：`README.md`、`03_開發路線圖.md`、`09_技術債務清單.md`、`04_更新日誌.md`。（`00`/`01` 留待後續 onboarding 文件補實時一併納入。）

### 階段三：模組化（為一年以上鋪路）

#### R0.2-09　`frustumCulled` 覆蓋面稽核　`[現在做]`　對應 TD-10
- **目的**：列出所有玩家 mesh 建立點，逐一確認 `frustumCulled=false`，把這個反覆性 bug 一次清乾淨。
- **影響範圍**：`EsportsGame.jsx`、`EsportsFPS3D.jsx`、`MobaView3D.jsx` 中所有玩家 mesh。
- **風險**：低。屬防禦性稽核，幾乎只增不改。
- **是否值得現在做**：**是**。範圍小、能根除一個反覆回歸的問題。
- **預估修改檔案**：`EsportsGame.jsx`、`EsportsFPS3D.jsx`、`MobaView3D.jsx`。

#### R0.2-10　拆分 monolith（受控、分批）　`[部分延後]`　對應 TD-09、TD-07
- **目的**：把 `EsportsGame.jsx`（9,401 行）依現有 Module 邊界拆成可獨立閱讀的檔案，向 B 血脈的模組化靠攏。
- **影響範圍**：`EsportsGame.jsx` 全檔，以及其與 `useLocalServer.js`/`GameView.jsx` 的銜接。
- **風險**：**高**。一次性大拆會大量打亂、難以回歸驗證，且沙盒不支援 import 會逼出雙模式維護。
- **是否值得現在做**：**部分**。本版只做**低風險的前置**——先抽資產（R0.2-05）、先補測試（R0.2-07）。**完整拆分延後到 v0.3**，前提是 R0.2-08 已確立 B 為主線、且 R0.2-07 的測試網已能守住行為。理由：在沒有測試與主線決策前硬拆，等於把最大的 Critical 風險疊在最不穩的地基上。
- **預估修改檔案**：（v0.3）`EsportsGame.jsx` 拆出的多個模組檔。

## 明確延後到 v0.3 以後（不在本版範圍）

| 項目 | 對應 TD | 延後理由 |

| 命令式 `useFrame`、FPS IIFE、重複小工具 | TD-13/14/15 | 已評為「接受現狀」，純粹性重寫不具效益。 |
