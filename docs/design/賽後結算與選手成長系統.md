# 賽後結算與選手成長系統（Sprint25）

## Result 與 Progress Transaction 的差別

**不要混淆這兩層。**

| | Result 契約 | Progress Transaction |
|---|---|---|
| 是什麼 | 「這場比賽發生了什麼」 | 「這場比賽要回寫什麼」 |
| 誰產生 | 遊戲引擎 / 快照 | 各遊戲的 Reward Adapter |
| 有幾種 | **兩種，各自獨立**：`BattleResult.v2`（MOBA）、`CsMatchResult.v1`（CS） | **一種**：`MatchProgressTransaction.v1` |
| 內容 | Hero / Dragon / Baron / Tower / QWER（MOBA）；回合 / ADR / KAST（CS） | 錢 / 粉絲 / 聲望 / 每位選手 XP・等級・天賦點 |

Sprint25 **沒有**合併兩個 Result，**沒有**建立萬用 MatchResult，**沒有**把兩種歷史
塞進同一個陣列。MOBA 歷史仍在 `seasonStore`，CS 歷史仍在 `profileStore.csHistory`。

新增的只是「回寫交易單」這一層共用語言。

## 資料流

```
BattleResult.v2 ──> mobaProgressAdapter ─┐
                                          ├─> MatchProgressTransaction.v1
CsMatchResult.v1 ─> csProgressAdapter ───┘              │
                                                        ▼
                                     applyMatchProgress（唯一發獎點）
                                                        │
                                     ┌──────────────────┴──────────────────┐
                                     ▼                                      ▼
                          profileStore（單一 set()）              Transaction Receipt
                          錢 / 粉絲 / 聲望                                  │
                          選手 xp / lv / talentPoints                       ▼
                          processedMatchTransactions        MOBA / CS Result Screen（只顯示，不重算）
```

## 結算邊界（重要）

**結算不掛在 Result Screen 上。**

| 模式 | 結算發生在哪 |
|---|---|
| MOBA | `useBattleFeed` 偵測引擎終局那一幀 |
| CS | `AppShell` 的 `CsMatchScreen.onFinish`（比賽完成邊界）→ `settleCsMatch` |

為什麼：Sprint23 的 CS 結算掛在 `CsResultScreen` 的 `useEffect`——
玩家若沒進 Result 畫面（直接返回 / 流程被打斷），獎勵就**永久漏發**。
Sprint25 兩邊都移到「比賽完成」的那一刻，Result Screen 只負責顯示 receipt。

## 順序（§10）

```
1. Result 產生
2. Result 契約驗證           ← 不合法就完全不寫入（不產生半套狀態）
3. 建立 Progress Transaction
4. applyMatchProgress        ← 唯一發獎點，單一 synchronous set()
5. 取得 receipt
6. 寫入 match history（附 transactionId）
7. Result Screen 顯示 receipt
```

**原子性**：zustand 無法真正做交易，但發獎的所有欄位（錢 / 粉絲 / 聲望 /
選手 XP / 等級 / 天賦點 / 冪等帳本）是在**同一個 `set()`** 裡一次寫完 →
不可能出現「錢寫了但 XP 沒寫」。history 在 receipt 之後才寫；
即使 history 寫入失敗，重跑同一張 transaction 也不會重複發獎（冪等）。

## 冪等策略（§5）

```
transactionId = `${mode}:${matchId}:progress-v1`
```

- **CS**：`matchId` 來自引擎（`CsMatchResult.matchId`），本來就有。
- **MOBA**：`BattleResult.v2` **沒有 matchId、也沒有時間戳**，而契約凍結不能加
  → `mobaMatchId(br)` 以比賽內容做 FNV-1a 決定性雜湊
  （勝方 / 時長 / 比分 / 金錢 / 塔數 / MVP / 10 位選手的 KDA・金錢・傷害）。

帳本：`profileStore.processedMatchTransactions[transactionId] = receipt`。

同一個 `transactionId` 再次套用 → 不加錢、不加粉絲、不加 XP、不發天賦點，
回傳既有 receipt 並標記 `alreadyApplied: true`。
⇒ React StrictMode 雙掛載、重整後重進 Result、返回再進入，都不會重複發獎。

## XP / 等級 / 天賦點規則

`platform/progress/playerLevel.js`（純函式，不依賴 React / Store）。

- `players[].xp` = **累積總 XP**（career total），不是「當級 XP」。
- `players[].lv` 一律由 `levelFromTotalXp(xp)` 導出 → 等級與 XP 不可能不一致。
- `xpRequiredForLevel(L) = 100 + (L−1) × 50`（線性遞增）
  → lv1→2 需 100（新秀一兩場就升）；lv38→39 需 1950（老將成長緩慢）。刻意的。
- 每升一級發 `TALENT_POINTS_PER_LEVEL = 1` 點天賦點。
- 保證：XP 不為負、等級不倒退、支援一次跨多級、整數比較不會因浮點誤差重複升級。
- **等級不受 potential 限制**（potential 限制的是 16 項「能力值」，見訓練系統）。

### 天賦點閉環

天賦點發到**選手身上**（`players[].talentPoints`）。
Dashboard 的「天賦」徽章 = 全隊未花費天賦點總和（衍生，不另存一份）。

⚠ 天賦「花費」介面尚未恢復（Legacy TalentModule 仍是佔位）→ 目前只進不出。

## 獎勵公式

`platform/progress/rewardFormulas.js`（唯一所在地；Result Screen / Store 都不得自己算）。

| 版本常數 | 值 | 來源 |
|---|---|---|
| `MOBA_REWARD_FORMULA_VERSION` | `moba-reward.v1` | Legacy `updateEconomy` **逐字重用** |
| `CS_REWARD_FORMULA_VERSION` | `cs-reward.v1` | Legacy `updateEconomy` **逐字重用** |
| `PLAYER_XP_FORMULA_VERSION` | `player-xp.v1` | Sprint25 新建（Legacy 沒有選手 XP） |

### 團隊獎勵（錢 / 粉絲）

兩種模式共用 Legacy `updateEconomy`，**未重新平衡**。差別只在 `marginF` 與 `streak`：

- MOBA：`marginF = win ? 3/8 : 0`（Legacy 非 CS 分支的語意）；streak 取自 seasonStore。
- CS：`marginF = |ourScore − enemyScore| / 8`；streak 取自 csHistory。

聲望（reputation）**沒有經 Legacy 驗證的公式** → 一律 `0`，不編造。

### 選手 XP

```
xp = round(base × perf) + (MVP ? 10 : 0)
base = 勝 50 / 負 20          （沿用 Legacy 團隊 xpGain 刻度，讓量級一致）
perf ∈ [0.75, 1.35]           （PERF_MIN / PERF_MAX，防雪球）
```

**不含 duration / roundCount** → 比賽拖長不會放大獎勵（§8）。

#### 角色公平（§9）— 這是本 Sprint 實測抓到的真問題

MOBA 的 `playerRating` 是**未正規化的原始分**
（`k×3 + a×1.5 − d×2 + dmg/800 + heal/1600 + gold/400`），重金錢與擊殺。
實測 fixture 中 carry 62 分 vs 輔助 10 分（差 6 倍）。
第一版公式讓 rating 佔一半權重 → **輔助直接被壓在下限，XP 只有隊均 70%**，
正是 §9 明文禁止的「Support 因 KDA 低而永遠無法成長」。

修正後：

```
MOBA perf = clamp(0.75 + participation × 0.5 + (rating/隊均 − 1) × 0.15, 0.75, 1.35)
CS   perf = clamp(0.5 × rating + 0.5 × (KAST/70), 0.75, 1.35)
```

- MOBA 以 **participation**（`(k+a)/隊伍總擊殺`，助攻同權）為主 → 對輔助公平；
  rating 只當 ±15% 的有限修正，保留 carry 的差別但不主宰結果。
- CS 的 rating 本來就以 1.0 為基準（已role-normalized）；再用 **KAST** 拉抬
  IGL / support（rating 天生低但回合貢獻高）。KAST 缺值 → 中性 1.0，不虛構貢獻。

## Persistence Migration（§12）

- `KEY` 維持 `esmo.profile.v1`（**不換 key、不清資料**），新增 `schemaVersion: 2`。
- `migratePlayer()`：
  - 舊資料（有 `lv` 無 `xp`）→ `xp = totalXpForLevel(lv)`，**保留原等級不倒退**。
    這是相容層，不是把「當級 XP」偷偷當成「總 XP」——因為根本沒有舊 XP 可轉換。
  - 已有 `xp` → `lv` 由 `xp` 重新導出，保證兩者永遠一致。
  - 損壞值（NaN / Infinity / 字串 / 負數）→ 安全降級，不讓 App 白畫面。
- 不清空 `csHistory`、不清空 MOBA history、不重置金錢 / 粉絲 / XP / 等級。

## Receipt 資料流

`receipt` = `applyMatchProgress` **實際套用後**的真實差額（不是預估值）：

```js
{ ok, applied, alreadyApplied, transactionId, matchId, mode,
  team: { money, fans, reputation, moneyBefore, moneyAfter, ... },
  players: [{ playerId, name, xpGained, previousLevel, newLevel, levelsGained, talentPointsGained, reasons }],
  totals: { xpGained, levelsGained, talentPointsGained } }
```

MOBA / CS Result Screen 共用 `ui/RewardReceiptPanel.jsx` 顯示它
→ **畫面顯示值與 Store 寫入值不可能分離**（這正是 Sprint25 要解的問題之一）。
`alreadyApplied` 為真時顯示「本場已結算（不重複發放）」，且不再播放成長動畫。

## 已知平衡風險

| 風險 | 現況 |
|---|---|
| CS 團隊獎金略高於 MOBA | fixture 實測 CS 勝 43 萬 vs MOBA 勝 33 萬（1.3×）。因 CS 的 `marginF` 用回合差（16:9 → 7/8），MOBA 固定 3/8。在 3× 容忍範圍內，**未重新平衡**（Legacy 公式凍結），列為觀察項。 |
| 聲望永遠 0 | 無經驗證的公式，不編造。 |
| 老將升級極慢 | lv38→39 需 1950 XP ≈ 25–40 場。刻意設計，但若體感太慢需調 `xpRequiredForLevel`。 |
| 天賦點只進不出 | 天賦花費介面（Legacy TalentModule）尚未恢復。 |

## 未來 Season / Sponsor 接入方式

- **Season**：`MatchProgressTransaction` 已有 `metadata.winner / score`。
  賽季積分只要新增一個 consumer 讀同一張 transaction，
  **不要**在 Result Screen 或 Store 裡再算一次戰績。
- **Sponsor**：贊助週收入 / 合約遞減應該做成獨立的「週結算」transaction
  （`mode: "week"`），走同一個 `applyMatchProgress` 冪等機制，
  而不是在賽後結算裡夾帶。
- **unlocks[]**：契約已預留（成就 / 解鎖），目前一律空陣列。
