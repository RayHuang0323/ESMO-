# 選手天賦與能力成長系統（Sprint27 MVP）

## 與 Legacy TalentModule 的差異（Audit 結論，刻意不同）

| | Legacy TalentModule | Sprint27 |
|---|---|---|
| 作用範圍 | **全隊**（allocateTalent 對整個 roster 加值） | **每位選手獨立** |
| 寫入方式 | **直接改 base stats**（`setRoster(stats[key]+1)`，不可逆且與訓練/potential 互相污染） | **derived 層**（base 永不被寫） |
| 點數 | 單一團隊池（React state） | S25 發放的 `players[].talentPoints`（每人獨立、持久化） |
| 規模 | 3 類 × 6 節點樹 | 4 類 × 3 節點 = 12（MVP 紅線） |

Legacy 的「真邏輯」只有 `allocateTalent`（扣點＋全隊 base+1），其餘是樹狀 UI。
S27 未複製任何 Legacy 元件。

## PlayerTalentState.v1

`src/platform/contracts/playerTalentState.js`

```js
{ version: "PlayerTalentState.v1",
  playerId,          // players[].id（"b1".."b5"/"r…"）——禁止名字/index
  availablePoints,   // = players[].talentPoints（S25 唯一點數來源，不另存一份）
  spentPoints,       // 一律由 definitions 重算（不信任持久層）
  ranks: { operation_1: 2, ... },
  updatedAt }
```

儲存位置：`profileStore.players[].talents = { ranks, spentPoints, updatedAt }`。
availablePoints 是讀取視圖；點數扣減直接作用在既有 `talentPoints` 欄位——
**不重置既有點數、不動 level / xp**。

## 12 個節點

每節點 maxRank 3、costPerRank 1；第 2 節需第 1 節 ≥2 級、第 3 節需第 2 節 ≥2 級。
每 rank 效果合計 ≤2 點（驗證腳本紅線）。滿配 36 點；單項能力加成上限 +6。

## Derived Stats

`src/platform/talents/playerDerivedStats.js`（純函式）：

```
base stats（訓練系統管理、potential 上限）
  + talent bonus（ranks × effects）
  = derived stats（clamp 1–99）
```

- `getPlayerDerivedStats(player)`：無天賦 → 逐鍵等於 base；不改輸入。
- `getStatLayers(player)`：`{base, talentBonus, derived}`（UI 分層顯示；
  未來加 training/condition 層只加欄位）。
- `withDerivedStats(player)`：stats 換成 derived 的複本——既有吃 `player.stats`
  的函式（calcPower/posFit/fpsRoster/adapter）零改動簽名即可用。
- 個性與狀態修正**不在此層**：calcPower 與 FPS 引擎 persStat 在自己那層套用
  （現況即如此，不重複套）。

## 購買服務（唯一入口）

純 reducer `applyTalentPurchase(player, talentId)` → `{ nextPlayer, receipt }`；
Store 包裝 `profileStore.purchasePlayerTalent({playerId, talentId})` 單一 set()。
檢查順序：player 存在 → talent 存在 → rank 上限 → 前置 → 點數（cost 只認
definition）→ 扣點 → 升 rank → spentPoints 重算 → receipt。
失敗 → 完全不寫入。UI 只顯示 receipt，不自行重算。

## MOBA / CS 注入

| 消費端 | 注入方式 | 效果 |
|---|---|---|
| CS 引擎（EsportsFPS3D） | `fpsRoster.toFpsRoster` 內 `toShortStats(getPlayerDerivedStats(p))` | **真的進引擎**——sim 的 `persStat` 直讀 stats[key]，天賦影響 rating/ADR/KAST 的輸入 |
| MOBA 戰術適性 | `screens/moba/tacticFit.js`（自 TacticScreen 抽出的純函式）讀 derived | 適性分隨天賦變（仍「僅供建議，不決定勝負」） |
| **MOBA 引擎（LogicEngine）** | **S28**：`mobaRosterAdapter.buildPlayerStatSlots`（依 playerId）→ `mobaPlayerStats.toEnginePlayerMods` → `useLocalServer` → `engine.configurePlayers` | **真的進引擎**——天賦平移 9 個行為作用點（撤退門檻/團戰參與/目標集結/推線深度/Gank 節奏/遊走/分推/入侵）。**不碰 power / tough / winner / gold / XP**。詳見 `MOBA選手能力注入.md` |
| 顯示層 | PlayerDetail（+N 標示）/ Roster / Team / Training / 天賦畫面 | 同一份 derived，各頁一致 |

無天賦時：derived === base（逐鍵）→ MOBA / CS baseline 與 Sprint26 一致。
中性能力（全 70）時 MOBA 引擎 mods 全 0 ⇒ **逐位元**回到 baseline。

⚠ **accuracy / learning 在 MOBA 引擎未映射**（誠實揭露，非遺漏）：引擎唯一「精準」的
表面是傷害，注入即違反紅線；learning 無跨場迴圈可掛。兩者在 **CS 引擎與戰術適性
仍完整生效**。（`mobaRosterAdapter.buildEngineSlots` 的 power/tough 推導**未被引擎採用**，
只服務 Legacy 沙盒 App.jsx——見 `MOBA選手能力注入.md` §6。）
（check_talent27 第 24/25 項 + 全套既有 verifier 綠燈）。

## 防誤點與不可重置

點節點 → 展開確認區（成本、能力 before→after、「⚠ 目前投入後不可重置」）
→ 再按「確認投入」才扣點。單次點擊不扣點。
本 Sprint 無正式重置系統；`__debugResetTalents` 僅供測試腳本
（驗證腳本確認它不出現在任何畫面）。

## Persistence / Migration

- `PROFILE_SCHEMA_VERSION 2 → 3`；同一個 localStorage key，不清資料。
- `migratePlayer` 追加 `talents: sanitizeTalents(p.talents)`：
  缺 → 空狀態；未知 talentId 忽略；rank 非數字→0、負→0、超上限→clamp；
  spentPoints 一律重算；talentPoints 負值/NaN → 0。
- 不重置 money / fans / xp / lv / history / csHistory / processedMatchTransactions。

## 數值上限與已知平衡風險

- 滿配 36 點 ≈ 36 次升級（新秀 lv1→~15 的全部點數）——短中期拿不滿。
- 單項 +6 對 calcPower 的影響 ≈ +1~2 綜合戰力（權重平均後），刻意溫和。
- CS：16 項全 +6 時 fpsOvr +6 → rating 輸入端明顯有感；但天賦點供給有限，
  實際單類 +3~6，屬可感知不碾壓。風險：多名滿配選手疊加 → 需在 Season
  引入對手成長對衡（未來 Sprint）。
- ~~MOBA 對戰暫不受影響（注入口未開）~~ → **S28 已關閉**：天賦經
  `configurePlayers` 進 MOBA 引擎行為層，CS / MOBA 不再不對稱。
- **MOBA 平衡風險（S28 實測 160 seeds）**：開啟能力層**不送勝率**
  （OFF 48% → 無天賦 49%）；天賦投資有回報（單人全滿 59%、全隊全滿 54%）；
  但**團隊天賦單投為負回報**（44%，−5pp，兩次取樣皆負）——本引擎勝負來自主堡
  血量（靠兵線與推塔破），「打團打得多」不必然轉化為勝利。這是引擎平衡屬性，
  不是天賦系統的 bug；**刻意不調權重去救勝率**。詳見 `MOBA選手能力注入.md` §8。
