### S29A：模擬規則集 / 時間 / 畫質（三條互不干擾的軸）

```
simTime          引擎世界時間。每 tick 固定推進 DT_SIM = 0.5 模擬秒。
                 ⇒ 與真實時間、瀏覽器 FPS、playbackRate **全部無關**。
playbackRate     只改「兩次 tick 之間的真實毫秒數」（tickMs = 500 / rate）。
                 dt 恆定 ⇒ **1×/2×/4× 不可能改變模擬結果**。
presentationTime 真實牆鐘。rAF 只用它算子幀進度 subT（0→1）供渲染內插。
                 渲染 FPS **不影響世界移動速度**（S29 §11 紅線）。
quality preset   battle/quality.js（low/medium/high）。只影響「怎麼畫」
                 （dpr / 後製 / 陰影 / FX 上限 / 光源），**絕不改變模擬結果**。
```

**模擬規則集**（`battle/moba/matchProgression.js` 的 `SIM_RULES`）：
- `v3`（**預設**，S29B1）= v2 全部 + **交戰狀態機**（參團黏性/撤退活得下來/追擊上限/
  復活鎖/DISENGAGE）+ **中立目標實體**（龍/巴龍真實 HP、6 座野怪營地、巴龍 buff 兵線）+
  **召喚師技能**（Flash 全員 / Smite 打野）+ **killContext** + 塔攻防（塔反擊英雄、
  人數優勢才可強攻）+ 死亡計時器成長。設計依據：`docs/design/MOBA交戰節奏與擊殺模型.md`、
  `MOBA中立目標與野區系統.md`、`MOBA召喚師技能系統.md`。
- `v2` = S29A 校準：本場 XP/等級（`mlv`/`mxp`，1–18）、真實移速、修好的小兵拆塔、
  **五個公平性修正**（`simultaneousCombat` / `twoPhaseTick` / `symmetricHot` /
  `nearestTarget` / `symmetricMinionCombat`）。保留供 baseline 對照
  （29B1 節奏門檻的檢定力對照組）。
- `v1` = 舊節奏（**壞的**），只保留供 baseline 對照：`new LogicEngine(seed, loadout, { rules: "v1" })`。

**S29B1 snapshot 附加欄位**（v3 才出現，舊快照形狀不變）：
`objectives[]`（中立目標：pos/alive/hp/respawn/killerTeam/participants）、
`players[].sp`（F/D 技能：ready/cd/reason）、`spellEvents`（最近 8 筆）、
`feed[].ctx`（killContext）。座標唯一來源 = `gameData.js`（`PITS`/`CAMPS`）
⇒ 主畫面 / Minimap / Replay 三處必然一致。

**S29B2 呈現層**：`snapshot.lanes[].bm/rm[].hp`（小兵血量，全規則集皆有；純觀測）；
Replay `frame.ob` = 目標 hp 值（0–1）；受擊/死亡可視化一律由 **prev/snapshot hp 差分**
推導（無假動畫）；響應式分歧唯一來源 `src/ui/useViewport.js`（useIsMobile）；
相機取景 `fitZoomFor`（BattleCameraController 匯出，CameraRig/跟隨共用）。
viewFx（死亡圈/火花）固定 14 格池，與引擎 fx 池分離。

**兩套等級絕不混用**：`mlv`/`mxp`（本場，終局丟棄）vs `lv`（英雄熟練，來自 Hero Progress
loadout）vs `profileStore.players[].lv/xp`（生涯，S25 賽後結算）。刻意不同名。

**戰術播報**（`battle/moba/tacticalComms.js`）：規則式、事件驅動、決定性、有 cooldown。
存在 `battleStore.comms`（與 `events`/`log` **分開** ⇒ `BattleResult.timeline` 契約不變）。

### MOBA 引擎的兩個注入層（S24 戰術 / S28 選手能力）

`LogicEngine` 有兩個**嚴格附加**的注入口。共同規則：**不呼叫 = 舊行為逐位元不變**；
knobs / mods 的形狀一律由**呼叫端**準備（引擎不 import 契約、不認得能力鍵名）；
只改「行為傾向 / 門檻 / 機率 / 節奏 / 路線」，**沒有任何傷害、勝率、金錢係數**。

| | S24 戰術層 | S28 選手能力層 |
|---|---|---|
| 開關 | `tacticOn` | `playerStatsOn` |
| 入口 | `configureMatch({blue,red,meta})` | `configurePlayers({blue,red,meta})` |
| 資料來源 | TacticScreen → `MobaTacticConfig.toEngineTactic` | profileStore.players → `getPlayerDerivedStats` → `mobaRosterAdapter.buildPlayerStatSlots` → `mobaPlayerStats.toEnginePlayerMods` |
| 對位 | side（blue/red） | **playerId**（`b1`–`b5`；非名字、非索引） |
| rng | 獨立 `rng2` 流（不污染主 `rng`） | **不新增任何抽樣**，只平移既有抽樣的門檻 |
| 中性 | `_neutralKnobs()` = 舊常數 | 能力全 70 分 ⇒ mods 全 0 / 倍率 1 |
| 證據 | `snapshot.tacticExec` | `snapshot.playerStatsExec`（每人撤退/團戰/目標計數） |

**呼叫順序**：`configurePlayers` 必須在 `configureMatch` **之前**——開局野區入侵在
`configureMatch` 當下擲骰，需要打野的 `invadeAdj`。唯一計算點在 `useLocalServer.start()`。

能力 → 引擎的 9 個作用點與 16 項能力映射表：見 `docs/design/MOBA選手能力注入.md`。
（`power` / `tough` **不受選手能力影響**——它直接乘進傷害式，注入即 damage multiplier。）
