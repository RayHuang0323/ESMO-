## 【P0・已修，長期釘住】players 陣列順序決定勝負

**這是正確性缺陷，不是平衡問題**：模擬結果曾取決於資料結構的擺放順序。
自 Sprint 04 起就存在，被「塔會瞬間融化」的粗暴節奏掩蓋。

實測（40 seeds，v1 = 缺陷版規則集）：把 `players` 陣列反轉，
**藍方勝率 55% → 6.7%**（位移 48 個百分點）。五個根因：

| # | 根因 | v2 修法 |
|---|---|---|
| ① | 傷害立即套用 ⇒ 先被迭代者先出手，被秒殺者連還手機會都沒有 | `simultaneousCombat`：傷害累積 → 一起套用 → 才判定死亡 |
| ② | 移動與交戰同一迴圈 ⇒ 先動的人用敵人**舊位置**判定接戰 | `twoPhaseTick`：全員先移動 → 再全員交戰 |
| ③ | `alive.find` ⇒ 一律集火「索引最小」的敵人 | `nearestTarget`：打**最近**的敵人 |
| ④ | 熱點取「陣列第一個」交戰者的鄰域 ⇒ 永遠繞著藍方長 | `symmetricHot`：取最密集鄰域、並列時 centroid |
| ⑤ | 小兵對戰只迭代藍方 ⇒ 紅兵被集火、系統性先死 | `symmetricMinionCombat`：雙方各自出手、同時結算 |
| ⑥ | `PITS.baron {33,32}` 不在雙方基地中垂線上 ⇒ 紅方近 2.45 單位 | 移到 `{33,34}`（差 0.20），連同 `OBSTACLES[0]` 水池 |

**長期防線**：`check_moba_runtime29` §24–29 的**正序／反序 invariant**
（傷害同時結算 / 小兵雙方迭代 / 最近目標選取 / 地圖對稱 / 整場勝率位移），
每一條都同時跑 v1 對照 —— **v1 的系統性翻轉必須被測出來**，否則測試等於沒檢定力。

⚠ **殘留（P1）**：v2 反轉陣列後**逐 seed 勝方仍可能改變**——rng 抽樣順序跟著 players
迭代順序走（特效骰、無戰術時的參戰骰）。那是**混沌**不是**偏差**（已證明無系統性
陣營優勢）。要做到「反轉後逐位元相同」需改 **per-player rng 流**（seed + player id 派生）；
會改變所有 v2 數值 ⇒ 需重新校準並重跑全部 verifier，**不在 29A 範圍**。

⚠ **殘留（P1）**：**換命時的擊殺 XP 歸屬仍看陣列順序**。`pendingHits` 先結算的那位攻擊者
拿得到擊殺 XP；後結算的那位此時已被標記 `dead` ⇒ `_addXp` 的 `p.dead` 守衛讓他拿不到。
實測同一換命場景：正序 b1 升 Lv2、反序改成 r1 升 Lv2。**不影響勝負**（雙方各記 1 殺，
`runtime29` §24 已驗），影響的是單一次換命的 XP。修法是讓 `_resolveKill` 的 XP 發放不受
攻擊者當下死活影響 —— 會改變所有 v2 數值 ⇒ 需重新校準並重跑全部 verifier，不在 29A 範圍。

⚠ **殘留（P1）**：`LANES` 折線、`WALLS`、`BUSHES` 仍不是精確 180° 鏡像
（例：藍方泉水到下路線頭 7.8 單位、紅方到另一端 11.2 單位，差 3.4）。
已修的是引擎機制真正依賴的部分（基地/泉水、塔與主堡路線參數、小兵出生、雙坑等距）。

## Sprint 28 殘留技術債（MOBA 能力注入）

1. **accuracy / learning 未映射到 MOBA 引擎**（誠實揭露，非遺漏）——引擎唯一

   「打團打得多」不必然轉化為勝利 ⇒ 團隊天賦（b3）藍勝率 44% vs 無天賦 49%。
   **這是引擎平衡屬性，不是注入層 bug**。刻意不調權重去救勝率（那是對著勝率
   調參）。正解：讓團戰勝利真的轉化為推進收益（Sprint 29 候選 1）。
3. **紅方（AI 對手）永遠中性**——無 profileStore 選手 ⇒ 全隊中性能力，是天然
   對照組，但也意味藍方能力平均高於 70 分時取得行為面優勢。要對稱需 AI 對手
   也有選手能力（＝ AI Teams）。

   缺「先發配置 → 引擎席位」的指派系統。
5. **power/tough 仍與選手無關**（刻意）——Phase 13 的 `calcMobaPower` 未採用，
