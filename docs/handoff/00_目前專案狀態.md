## Sprint 29B4 完成（Debug Recovery, Replay Fix and Clickability）

29B3 部署後實測四修（不改節奏/XP/Progress/Reward/gameData 座標）：

- **快速完成比賽（?debug=1 看不到）根因**：按鈕原本被收進手機 ⚙ 收合面板（showCtl），
  且 `isDebugMode` 只讀 `location.search`（Pages 帶 hash 的網址讀不到）。修法：按鈕改
  **測試模式常駐可見**（脫離 ⚙ 收合）；`isDebugMode` 改讀 search+hash 並於認定後
  寫入 `localStorage.esmo_debug` 持久化。行為仍是同引擎推進到終局、走既有 Result/
  發獎（冪等）/Replay 流程。
- **10 英雄部分可點根因**：`HeroDetailPanel` 舊碼 `if (!hero) return null`——無 HeroProgress
  紀錄的英雄（對手方、或 heroId 不在已存 progress 集合）點了回傳 null＝沒反應。
  修法：無紀錄時用 `emptyHero()` 佔位，面板照常開啟＋「尚無成長紀錄」註記 ⇒ 10/10 一致。
- **Replay 無法觀看**：buffer/finalize 經 Node 實測本就健康（自然+快速完成皆產生有效
  replay、matchId 同源）；本 Sprint 補 `MobaReplayScreen` **白畫面防護**（frame/meta
  缺欄位一律安全預設，舊格式不崩），並保留誠實的「無法重播」停用狀態。
- **塔常駐光移除**：crystal 常態 emissive 降到主堡 0.14 / 塔 0.06（低於 Bloom 門檻 ⇒
  無常駐光暈），受擊才 +1.6 脈衝、摧毀有爆點；主堡 PointLight 3.5→2.0；**未新增 PointLight**。

**Audit（Node 實測）**：LAG **不在邏輯層**（tick 只佔預算 0.03%）；本場英雄等級系統

**⚠【P0】最重要的發現：`players` 陣列順序決定勝負**——這是**正確性缺陷**，不是平衡問題：
模擬結果取決於資料結構的擺放順序。反轉陣列即翻轉勝負（可重現的量化基準：v1 規則集、
40 seeds，藍方勝率 **55% → 6.7%**）。五個順序偏差（先手扣血 / 移動交戰同迴圈 /

**舊 regress 的「藍勝 9/20」是假平衡**——bug 互相抵銷。修正後藍勝 **12/20**（60 seeds：31/60），
反轉陣列不再產生系統性偏差。
